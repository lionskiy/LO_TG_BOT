> **Дата:** 2026-02-07

## Как пользоваться

## 1. База данных и модель

- [ ] **1.1** Таблица сотрудников
  - [ ] 1.1.1 Определить имя таблицы (`employees` или `hr_employees`) и размещение в той же SQLite, что и telegram_settings / tool_settings.
  - [ ] 1.1.2 Реализовать схему: все поля по SPEC разд. 2.1 (id, personal_number, full_name, email, jira_worker_id, position, mvz, supervisor, hire_date, fte, dismissal_date, birth_date, mattermost_username, is_supervisor, is_delivery_manager, team, created_at, updated_at).
  - [ ] 1.1.3 Индексы: unique по `personal_number`, по `email`, по `jira_worker_id`.
- [ ] **1.2** Создание таблицы
  - [ ] 1.2.1 Миграция (Alembic) или скрипт инициализации при первом обращении к БД / при старте приложения.
  - [ ] 1.2.2 Проверка: таблица создаётся и доступна из кода плагина.

## 2. Плагин hr_service (структура и регистрация)

- [ ] **2.1** Манифест и регистрация
  - [ ] 2.1.1 Создать каталог `plugins/hr_service/`.
  - [ ] 2.1.2 Файл `plugin.yaml`: id, name, version, description, enabled; один инструмент (например, `name: hr`) с параметром `action` и описанием параметров по SPEC разд. 4.2.
  - [ ] 2.1.3 Секция `settings`: jira_url, api_token (и при необходимости jira_email / jira_username).
  - [ ] 2.1.4 Плагин загружается через общий Plugin Loader; отображается в админке «Инструменты».
- [ ] **2.2** Handler и dispatch
  - [ ] 2.2.1 Файл `handlers.py`: один handler (например, `hr_dispatch`), принимает `action` и аргументы.
  - [ ] 2.2.2 По `action` вызываются: get_employee, list_employees, search_employees, update_employee, import_employees.
  - [ ] 2.2.3 Контракт ошибок: единый формат (например, "Error: …" или JSON с полем error) по DOCUMENTATION_AUDIT.

## 3. Функции плагина (чтение и обновление)

- [ ] **3.1** get_employee
  - [ ] 3.1.1 Поиск по ФИО, табельному номеру или email.
  - [ ] 3.1.2 Возврат данных одной записи или сообщение об отсутствии.
- [ ] **3.2** list_employees
  - [ ] 3.2.1 Список с опциональными фильтрами (МВЗ, команда, только руководители, только деливери-менеджеры).
  - [ ] 3.2.2 Пагинация при необходимости (на усмотрение реализации).
- [ ] **3.3** search_employees
  - [ ] 3.3.1 Поиск по имени, подразделению, должности и т.д.
- [ ] **3.4** update_employee
  - [ ] 3.4.1 Обновление полей записи (ставка, дата увольнения, is_supervisor, is_delivery_manager, team, mvz, supervisor и т.д.).
  - [ ] 3.4.2 Поиск по ФИО или табельному номеру; при однофамильцах — возможность уточнения (возврат списка кандидатов).
  - [ ] 3.4.3 Проверка «сервисный админ» при вызове из бота (по Telegram ID) — только для этой функции и import_employees; в API админки проверка доступа к админке.

## 4. Импорт из Excel

- [ ] **4.1** Парсинг файла
  - [ ] 4.1.1 Поддержка форматов .xlsx и .xls.
  - [ ] 4.1.2 Два листа с именами «ДДЖ» и «Инфоком»; маппинг колонок по SPEC разд. 3.3 (в т.ч. однозначное различение «Табельный №» и «Табельный номер» с ФИО на листе «Инфоком»).
  - [ ] 4.1.3 Объединение данных по полю personal_number (один номер — одна запись).
- [ ] **4.2** Валидация и запись в БД
  - [ ] 4.2.1 Проверка дубликатов табельного номера внутри файла (в обоих листах или дважды на одном). При дубликате — ошибка с расшифровкой (какой номер, какие листы/строки); импорт не выполнять.
  - [ ] 4.2.2 Сверка с БД: существующие по personal_number — пропуск; новые — вставка с дефолтами (fte=1, jira_worker_id=NULL, is_supervisor/is_delivery_manager=false, team=NULL, mattermost_username=email и т.д.).
- [ ] **4.3** Обратная связь
  - [ ] 4.3.1 Сообщение пользователю: сколько добавлено; если новых ≤10 — перечислить фамилии; если больше — только число. При ошибках — факт и количество.

## 5. Дообогащение из Jira

- [ ] **5.1** Клиент Jira
  - [ ] 5.1.1 Получение настроек плагина (jira_url, api_token; при необходимости jira_email/jira_username).
  - [ ] 5.1.2 Запрос `GET /rest/api/2/user?username={username}`; из ответа — поле `key` (jira_worker_id). Username — из email (часть до `@`) или из настроек.
- [ ] **5.2** Интеграция в импорт
  - [ ] 5.2.1 После успешного импорта для всех новых сотрудников с пустым jira_worker_id — запрос к Jira, запись `key` в поле `jira_worker_id`.
  - [ ] 5.2.2 При ошибке для части сотрудников — не блокировать импорт; записать обогащённых; в отчёте или логе указать количество не обогащённых.
- [ ] **5.3** Тест подключения (опционально)
  - [ ] 5.3.1 Функция теста (например, GET user) для кнопки «Проверить» в админке «Инструменты» (POST /api/tools/{name}/test).

## 6. Интеграция с Telegram-ботом

- [ ] **6.1** Передача файла в плагин
  - [ ] 6.1.1 В ядре/боте: при получении вложения (документ) — скачивание файла (getFile), сохранение во временный файл (или передача bytes).
  - [ ] 6.1.2 Вызов плагина hr_service, action=import_employees, с путём к файлу или содержимым (контракт по SPEC разд. 8). Ограничение размера и удаление временного файла — в ядре.
  - [ ] 6.1.3 Если в сообщении только файл (или нейтральный текст) — интерпретировать как импорт и передать в плагин.
- [ ] **6.2** Проверка прав при действиях в боте
  - [ ] 6.2.1 Перед import_employees и update_employee: проверка Telegram ID отправителя по списку service_admins. Если не админ — отказ без выполнения.
- [ ] **6.3** Команды на изменение через LLM
  - [ ] 6.3.1 Фразы вида «У Беспаловой ставка 0,5», «Беспалова уволена с 01.02.2025» обрабатываются вызовом инструмента hr с action=update_employee и соответствующими аргументами (LLM извлекает сущность, поле, значение).

## 7. API для админки «Работа с БД»

- [ ] **7.1** Эндпоинты (защита: тот же механизм, что и для админки, например X-Admin-Key)
  - [ ] 7.1.1 `GET /api/hr/employees?view=all|supervisors|delivery_managers` — список сотрудников для трёх представлений.
  - [ ] 7.1.2 `GET /api/hr/employees/{id}` — одна запись (опционально, если нужно для модального редактирования).
  - [ ] 7.1.3 `PATCH /api/hr/employees/{id}` — частичное обновление полей (редактирование в ячейках).
  - [ ] 7.1.4 `POST /api/hr/import` — загрузка файла (multipart/form-data); логика как в разд. 3–4 SPEC; ответ — JSON с результатом (сколько добавлено, ошибки).
- [ ] **7.2** Роутер и подключение
  - [ ] 7.2.1 Вынести маршруты в отдельный роутер (например, `api/hr_router.py` или в существующую структуру) и подключить в `app.py`.

- [ ] **8.1** Размещение и навигация
  - [ ] 8.1.1 Пункт меню «Работа с БД» в админке.
  - [ ] 8.1.2 Страница по URL `/admin/#db`.

  - [ ] 8.4.1 Кнопка «Импорт» на странице; выбор файла .xlsx/.xls; отправка на POST /api/hr/import; отображение результата (сколько добавлено, ошибки при наличии).
